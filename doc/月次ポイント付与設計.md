# 毎月定期ポイント配布機能の設計

## 概要
毎月1日午前0時（JST）に、対象ユーザーに対して自動でポイントを付与する機能。
対象ユーザーはprivilegedユーザー。pk=privileged-users,sk=privileged-usersで取得
ユーザー個別設定で0ポイントに設定されている場合は配布しない。

## DB設計

### 新規テーブル項目

#### 1. ポイント配布設定（point-allocation-config）
| PK | SK | 説明 |
|----|----|----|
| config#point-allocation | default | デフォルトポイント配布設定 |
| config#point-allocation | user#{userid} | ユーザー個別設定 |

**属性:**
- pk: `config#point-allocation`
- sk: `default` または `user#{userid}`
- dtype: `point-allocation-config`
- monthlyPoints: number（月次配布ポイント数）
- updatedAt: 更新日時

#### 2. ポイント配布履歴（point-allocation-history）
| PK | SK | 説明 |
|----|----|----|
| pt-alloc-hist#{yyyy-mm} | {userid}#{timestamp} | 月次配布履歴 |

**属性:**
- pk: `pt-alloc-hist#{yyyy-mm}`
- sk: `{userid}#{timestamp}`
- userId: 対象ユーザーID
- allocatedPoints: 配布ポイント数
- expirationDate: ポイント有効期限（月末24時）
- errorMessage: エラーメッセージ（失敗時のみ）
- createdAt: 実行日時

### 既存テーブル活用

#### ポイント残高（pt）
定期配布時も既存のポイント残高テーブルを使用：
- pk: `pt#{userid}`
- sk: ポイント有効期限（月末24時）
- balance: ポイント残高
- dtype: `pt`
- createdAt: 作成日時
- updatedAt: 更新日時

#### ポイント残高更新ログ（ptlog）
定期配布時のログも記録（既存の形式を使用）：
- pk: `ptlog#{userid}`
- sk: `{有効期限}#{作成日時}`
- balance: 更新後の残高
- additionalPoints: 配布ポイント数
- createdAt: 作成日時


## Lambda関数設計

### 定期実行用Lambda（scheduled-point-allocation）
- **トリガー**: EventBridge Scheduler（毎月1日午前0時JST）
- **ファイルパス**: `/backend-cdk/lambda/scheduler/scheduled-point-allocation.py`
- **テスト機能**: `custom_expiration_days`または`custom_expiration_minutes`パラメータで有効期限を指定可能
- **処理内容**:
  1. デフォルト設定を取得（pk=config#point-allocation, sk=default）
  2. privilegedユーザーリストを取得（pk=privileged-users, sk=privileged-users）
  3. 各ユーザーの個別設定を確認（pk=config#point-allocation, sk=user#{userid}）
  4. 0ポイント設定のユーザーはスキップ（履歴を記録）
  5. 冪等性チェック: 当月すでに配布済みならスキップ（`is_already_allocated()`）
  6. ポイント配布実行（`common.add_points()`）
  7. 履歴記録（pt-alloc-hist#{yyyy-mm}）

### 冪等性制御
複数環境（aya-dev, bsn-proto-prod等）でDynamoDBを共有しているため、二重配布を防止する仕組み。

- **チェック方法**: `pt-alloc-hist#{yyyy-mm}`を`{userid}#`プレフィックスでクエリ
- **判定**: `errorMessage`属性がないレコードが存在すれば「配布済み」と判定しスキップ
- **エラー時**: チェック自体が失敗した場合は安全側に倒して「配布済み」とみなす（スキップ）
- **注意**: 0ポイントスキップ時も`errorMessage`なしで履歴が記録されるため、後からデフォルトポイントを設定しても当月中は配布されない

### EventBridge Scheduler設定
- **サービス**: AWS EventBridge Scheduler（`aws_scheduler.CfnSchedule`）
- **実行スケジュール**: `cron(0 0 1 * ? *)`（毎月1日0:00 JST）
- **タイムゾーン**: `Asia/Tokyo`
- **CDK設定**: `cdk_chatgpt_clone_stack_ap.ts`に追加済み
- **IAMロール**: `scheduler.amazonaws.com`がLambdaを呼び出すためのロールを別途作成


## ポイント有効期限の計算

有効期限は「日数」で計算し、`common.add_points()`内で`now + timedelta(days=effective_days)`として期限日時を算出する。

```python
def calculate_effective_days(custom_days=None, custom_minutes=None):
    """有効期限までの日数を計算

    Args:
        custom_days: テスト用のカスタム有効期限日数
        custom_minutes: テスト用のカスタム有効期限分数（custom_daysより優先）

    Returns:
        有効期限日数（整数）
    """
    if custom_minutes is not None:
        return max(1, int(custom_minutes / 1440))  # 1440分 = 1日
    elif custom_days is not None:
        return custom_days
    else:
        # 本番用: 月末までの日数を計算（当日を含む）
        now = datetime.now(timezone("Asia/Tokyo"))
        last_day = calendar.monthrange(now.year, now.month)[1]
        remaining_days = last_day - now.day + 1
        return remaining_days
```

**例**: 4月1日実行 → `last_day=30`, `remaining_days=30` → 有効期限は`now + 30日` = 5月1日

## 初期設定・テスト方法

### デフォルト設定の作成
```python
# DynamoDBに直接登録
{
  "pk": "config#point-allocation",
  "sk": "default", 
  "dtype": "point-allocation-config",
  "monthlyPoints": 10000,
  "updatedAt": "2025-01-08 00:00:00"
}
```

### テスト実行
```bash
# 7日後を有効期限にする場合
aws lambda invoke \
  --function-name ${PROJECT_NAME}-scheduled-point-allocation \
  --cli-binary-format raw-in-base64-out \
  --payload '{"custom_expiration_days": 7}' \
  response.json

# 1分後を有効期限にする場合（分単位テスト用）
aws lambda invoke \
  --function-name ${PROJECT_NAME}-scheduled-point-allocation \
  --cli-binary-format raw-in-base64-out \
  --payload '{"custom_expiration_minutes": 1}' \
  response.json
```
